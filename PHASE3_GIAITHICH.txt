VẤN ĐỀ CỐT LÕI: TẠI SAO NAIVE RAG CHƯA ĐỦ?
==================================================

M1 (Naive RAG) chỉ dùng Dense Retrieval — tìm chunks có NGHĨA gần với
câu hỏi. Nhưng dense retrieval có điểm yếu:

    Câu hỏi: "GDP của Việt Nam năm 2023 là bao nhiêu?"
    Trong document có: "tổng sản phẩm quốc nội Việt Nam 2023 đạt 430 tỷ USD"

    Dense: "GDP" và "tổng sản phẩm quốc nội" có nghĩa giống nhau
           → embedding gần nhau → tìm được ✓

    Nhưng nếu câu hỏi dùng đúng từ khóa đặc biệt:
    "Nghị định 13/2023/NĐ-CP quy định điều gì?"
    Dense: "nghị định" "13" "2023" là từ đặc biệt, embedding không phân biệt tốt
    → có thể tìm nhầm nghị định khác ✗

    Ngược lại BM25 (keyword matching) sẽ tìm đúng "13/2023/NĐ-CP" vì
    nó match exact từ khóa.

→ Giải pháp: kết hợp cả hai (Hybrid Search)

FILE 1 — src/retrieval/bm25.py
==================================================

MỤC ĐÍCH: Tìm kiếm dựa trên từ khóa (keyword matching) sử dụng BM25.

CODE:
    def _load_chunks_from_chroma(chroma_dir: str) -> list[Document]:
        db_path = os.path.join(chroma_dir, "chroma.sqlite3")
        conn = sqlite3.connect(db_path)
        cur = conn.cursor()
        cur.execute("SELECT string_value FROM embedding_metadata WHERE key='chroma:document'")
        rows = cur.fetchall()
        conn.close()
        return [Document(page_content=r[0]) for r in rows if r[0]]

    class BM25Retriever:
        def __init__(self):
            self.docs = _load_chunks_from_chroma(CHROMA_DIR)
            tokenized = [doc.page_content.lower().split() for doc in self.docs]
            self.bm25 = BM25Okapi(tokenized)

        def retrieve(self, query: str, k: int = TOP_K) -> list[Document]:
            tokens = query.lower().split()
            scores = self.bm25.get_scores(tokens)
            top_indices = sorted(range(len(scores)), key=lambda i: scores[i], reverse=True)[:k]
            return [self.docs[i] for i in top_indices]

GIẢI THÍCH:

    _load_chunks_from_chroma():
    - BM25 cần toàn bộ corpus (tất cả chunks) để tính điểm
    - Không dùng lại ChromaDB API vì ChromaDB không hỗ trợ trả về toàn bộ docs dễ dàng
    - Đọc thẳng từ file SQLite: data/chroma_db/chroma.sqlite3
    - Table embedding_metadata lưu text của mỗi chunk với key='chroma:document'
      (Lưu ý: key phải là 'chroma:document' không phải 'document' — đây là
       convention của ChromaDB, phát hiện qua debug SQLite)
    - Kết quả: 831 Document objects chứa toàn bộ chunks đã index

    BM25Retriever.__init__():
    - tokenized: chuyển mỗi chunk thành list từ (lowercase, split by space)
      VD: "Tháp Eiffel cao 324 mét" → ["tháp", "eiffel", "cao", "324", "mét"]
    - BM25Okapi(tokenized): build index BM25 từ toàn bộ 831 chunks
      BM25 = Best Match 25, thuật toán ranking phổ biến nhất trong Information Retrieval
      (dùng trong Elasticsearch, Lucene, nhiều search engine lớn)

    BM25 hoạt động như thế nào?
    BM25 tính score cho mỗi document dựa trên:
      - Term Frequency (TF): từ xuất hiện nhiều lần trong doc → score cao hơn
        Nhưng có giới hạn (saturation) — từ xuất hiện 100 lần không tốt gấp
        10 lần so với từ xuất hiện 10 lần
      - Inverse Document Frequency (IDF): từ hiếm gặp (ít doc chứa) → quan trọng hơn
        "tháp" xuất hiện nhiều doc → IDF thấp
        "13/2023/NĐ-CP" chỉ trong 1 doc → IDF cao
      - Document Length Normalization: doc dài không được ưu tiên hơn doc ngắn
        chỉ vì chứa nhiều từ hơn

    retrieve(query, k=4):
    - Tokenize câu hỏi: "Tháp Eiffel cao bao nhiêu" → ["tháp", "eiffel", "cao", "bao", "nhiêu"]
    - get_scores(): tính BM25 score cho tất cả 831 chunks
    - Sắp xếp giảm dần, lấy top-k
    - Trả về list Document (không có embedding, chỉ có text)

    So sánh với Dense:
    BM25: nhanh hơn (không cần embed), tốt với từ khóa đặc thù, số liệu, tên riêng
    Dense: hiểu ngữ nghĩa, tốt với paraphrase, câu hỏi dùng từ đồng nghĩa

FILE 2 — src/retrieval/hybrid.py
==================================================

MỤC ĐÍCH: Kết hợp BM25 và Dense bằng thuật toán RRF (Reciprocal Rank Fusion).

CODE:
    def rrf_fusion(ranked_lists: list[list[Document]], k: int = RRF_K) -> list[Document]:
        scores: dict[str, float] = {}
        doc_map: dict[str, Document] = {}
        for ranked in ranked_lists:
            for rank, doc in enumerate(ranked):
                key = doc.page_content
                scores[key] = scores.get(key, 0.0) + 1.0 / (k + rank + 1)
                doc_map[key] = doc
        sorted_keys = sorted(scores, key=lambda x: scores[x], reverse=True)
        return [doc_map[k] for k in sorted_keys]

    class HybridRetriever:
        def __init__(self):
            self.dense = DenseRetriever()
            self.bm25 = BM25Retriever()

        def retrieve(self, query: str, k: int = TOP_K) -> list[Document]:
            dense_docs = self.dense.retrieve(query, k=k)
            bm25_docs = self.bm25.retrieve(query, k=k)
            fused = rrf_fusion([dense_docs, bm25_docs])
            return fused[:k]

GIẢI THÍCH:

    Vấn đề khi gộp 2 danh sách:
    Dense trả về scores kiểu cosine similarity (0.0 → 1.0)
    BM25 trả về scores kiểu TF-IDF (0 → vài chục, tùy corpus)
    → Không thể cộng trực tiếp vì khác đơn vị, khác scale

    Giải pháp — RRF (Reciprocal Rank Fusion):
    Không dùng score, chỉ dùng RANK (thứ hạng) của mỗi doc trong mỗi list.

    Công thức RRF:
        Score(doc) = Σ  1 / (RRF_K + rank_i + 1)
                    i
    Trong đó:
        rank_i  = vị trí của doc trong danh sách thứ i (bắt đầu từ 0)
        RRF_K   = 60 (hằng số, lấy từ paper gốc 2009)
        +1      = để tránh chia cho 0 khi rank=0

    Ví dụ cụ thể (RRF_K=60, TOP_K=4):

        Dense list:   [A, B, C, D]   → rank: A=0, B=1, C=2, D=3
        BM25 list:    [C, A, E, F]   → rank: C=0, A=1, E=2, F=3

        Score(A) = 1/(60+0+1) + 1/(60+1+1) = 1/61 + 1/62 ≈ 0.0164 + 0.0161 = 0.0325
        Score(B) = 1/(60+1+1) + 0           = 1/62         ≈ 0.0161
        Score(C) = 1/(60+2+1) + 1/(60+0+1) = 1/63 + 1/61 ≈ 0.0159 + 0.0164 = 0.0322
        Score(E) = 0           + 1/(60+2+1) = 1/63         ≈ 0.0159
        Score(D) = 1/(60+3+1) + 0           = 1/64         ≈ 0.0156

        Final ranking: A (0.0325) > C (0.0322) > B (0.0161) > E (0.0159) > D (0.0156)

    Ý nghĩa:
    - A xuất hiện ở #1 Dense và #2 BM25 → score cao nhất → kết quả tốt nhất
    - C xuất hiện ở #3 Dense và #1 BM25 → score gần bằng A → cả 2 phương pháp đều thấy quan trọng
    - B chỉ có trong Dense → score thấp hơn A và C
    - E chỉ có trong BM25 → xuất hiện trong kết quả cuối (điều Dense bỏ sót)

    Tại sao RRF_K=60?
    Paper gốc (Cormack et al. 2009) đề xuất k=60 như default.
    k lớn → ít nhạy cảm với thứ hạng đầu (smooth hơn)
    k nhỏ → ưu tiên mạnh vào top ranks

    rrf_fusion() dùng doc.page_content làm key (string):
    - Nếu cùng 1 chunk xuất hiện trong cả 2 list → được tích lũy score
    - Nếu chunk chỉ trong 1 list → chỉ có score từ 1 nguồn

    HybridRetriever.retrieve():
    1. Chạy Dense → top-4 chunks (semantic)
    2. Chạy BM25 → top-4 chunks (keyword)
    3. RRF fusion → merge + rerank → top-4 chunks cuối

MỤC ĐÍCH: Pipeline M2 hoàn chỉnh — thay DenseRetriever bằng HybridRetriever.

CODE:
    class HybridRAG:
        def __init__(self):
            self.retriever = HybridRetriever()
            self.llm = LLMWrapper()

        def run(self, question: str, k: int = TOP_K) -> dict:
            docs = self.retriever.retrieve(question, k=k)
            context = "\n\n".join(d.page_content for d in docs)
            prompt = M1_PROMPT.format(context=context, question=question)
            answer = self.llm.generate(prompt)
            return {
                "question": question,
                "answer": answer,
                "contexts": [d.page_content for d in docs],
            }

GIẢI THÍCH:

    M2 gần như giống hệt M1 — chỉ khác 1 điểm:
        M1: self.retriever = DenseRetriever()
        M2: self.retriever = HybridRetriever()

    Phần còn lại (prompt, LLM, output format) giữ nguyên.
    → Điều này cho phép so sánh thuần túy ảnh hưởng của retrieval strategy.

    Luồng dữ liệu M2:
        "Câu hỏi?"
            ↓
        HybridRetriever
            ├── DenseRetriever → top-4 (semantic)
            └── BM25Retriever  → top-4 (keyword)
                        ↓
                    RRF fusion → top-4 (combined)
            ↓
        M1_PROMPT.format(context, question)
            ↓
        LLMWrapper (Groq llama-3.3-70b-versatile)
            ↓
        "Câu trả lời"

    Nếu chỉ search 1 lần với câu hỏi gốc → có thể bỏ sót các chunk dùng

    → mỗi dòng là 1 variant, dễ parse bằng splitlines()

FILE 5 — src/methods/m3_fusion.py
==================================================

MỤC ĐÍCH: Pipeline M3 — thêm bước query expansion trước khi search.

CODE:
    N_VARIANTS = 3

    class FusionRAG:
        def __init__(self):
            self.retriever = HybridRetriever()
            self.llm = LLMWrapper()

        def _expand_query(self, question: str) -> list[str]:
            prompt = QUERY_EXPANSION_PROMPT.format(n=N_VARIANTS, question=question)
            raw = self.llm.generate(prompt)
            variants = [line.strip() for line in raw.strip().splitlines() if line.strip()]
            return variants[:N_VARIANTS]

        def run(self, question: str, k: int = TOP_K) -> dict:
            variants = self._expand_query(question)
            all_queries = [question] + variants

            all_ranked = [self.retriever.retrieve(q, k=k) for q in all_queries]

            fused = rrf_fusion(all_ranked)[:k]

            context = "\n\n".join(d.page_content for d in fused)
            prompt = M1_PROMPT.format(context=context, question=question)
            answer = self.llm.generate(prompt)
            return {
                "question": question,
                "answer": answer,
                "contexts": [d.page_content for d in fused],
                "query_variants": variants,
            }

GIẢI THÍCH:

    _expand_query(question):
    - Gọi LLM với QUERY_EXPANSION_PROMPT → nhận raw text
    - splitlines(): tách theo dòng (mỗi dòng 1 variant)
    - filter(line.strip()): bỏ dòng trống
    - [:N_VARIANTS]: lấy tối đa 3 variants (phòng LLM sinh thêm)

    Ví dụ output của _expand_query:
        Input:  "Tháp Eiffel được xây dựng để kỷ niệm sự kiện gì?"
        Output: ["Mục đích xây dựng Tháp Eiffel là gì?",
                 "Tháp Eiffel ra đời nhân dịp sự kiện nào?",
                 "Lý do nào dẫn đến việc xây dựng Tháp Eiffel?"]

    run(question):
    Bước 1 — Query Expansion (LLM Call #1):
        variants = _expand_query(question)
        all_queries = [question] + variants
        → all_queries có 4 câu: 1 gốc + 3 variants

    Bước 2 — Multi-query Hybrid Search:
        all_ranked = [self.retriever.retrieve(q, k=4) for q in all_queries]
        → Chạy HybridRetriever cho TỪNG query trong all_queries
        → 4 queries × top-4 results = 4 list, mỗi list 4 chunks
        → Tổng cộng có thể lên đến 16 chunks khác nhau

    Bước 3 — RRF lần 2 (RRF×2):
        fused = rrf_fusion(all_ranked)[:4]
        → Gộp 4 ranked lists bằng RRF một lần nữa
        → Chunk nào xuất hiện trong nhiều query result → score cao → ưu tiên
        → Lấy top-4 cuối cùng

        Ví dụ: chunk về "Tháp Eiffel" xuất hiện trong kết quả của cả 4 queries
        → được tích lũy RRF score từ 4 nguồn → rank cao nhất

    Bước 4 — Generate (LLM Call #2):
        prompt = M1_PROMPT.format(context, question)
        answer = self.llm.generate(prompt)

    M3 dùng 2 LLM calls/câu hỏi (gấp đôi M1/M2):
        Call 1: sinh query variants (~200-400 tokens)
        Call 2: generate answer (~500-1000 tokens)
    → Tốn quota gấp đôi → cần chú ý khi chạy eval

    output thêm "query_variants": lưu lại để phân tích sau
        VD xem LLM expand query như thế nào, có đa dạng không

    Tại sao gọi rrf_fusion() lần 2 thay vì concat rồi dedup?
    RRF ưu tiên chunk được nhiều query đồng thuận:
        Nếu chunk A xuất hiện ở top của cả 4 query result
        → score = 4 × (1/61) ≈ 0.066 → rất cao
        Nếu chunk B chỉ xuất hiện trong 1 query result ở top
        → score = 1 × (1/61) ≈ 0.016 → thấp hơn nhiều
    → Chunks "cross-query relevant" được ưu tiên

M2 — Hybrid RAG:

        ├──► DenseRetriever  → [A, B, C, D] (semantic)

    "Câu hỏi gốc?"
        ↓ LLM Call #1 (expand)
    ["Variant 1?", "Variant 2?", "Variant 3?"]
        ↓
    all_queries = ["Câu hỏi gốc?", "Variant 1?", "Variant 2?", "Variant 3?"]
        ↓ HybridRetriever × 4
    [list1=[A,B,C,D], list2=[A,E,C,F], list3=[A,B,G,H], list4=[A,C,I,J]]
        ↓ rrf_fusion() lần 2 (RRF×2)
    [A, C, B, E]   ← A xuất hiện top-1 trong cả 4 lists → score cao nhất
        ↓
    M1_PROMPT + LLM Call #2 → "Câu trả lời"

PHÂN TÍCH KẾT QUẢ:

    1. Context Hit Rate M2 = M3 = 0.91:
       Cả hai đều dùng Hybrid Search → retrieval tốt như nhau
       Tăng 8.5% so M1 (0.825) chứng tỏ BM25 bổ sung nhiều

    2. Context Recall M3 > M2 (0.875 > 0.714):
       Multi-query tìm được nhiều góc độ context hơn
       → Ground truth được bao phủ nhiều hơn trong retrieved contexts

    3. Faithfulness M2 > M3 (0.733 > 0.529):
       M3 retrieve nhiều context đa dạng → LLM có xu hướng pha trộn
       thông tin từ nhiều nguồn → tăng hallucination
       M2 context tập trung hơn → LLM bám sát hơn

    4. Answer Relevancy M2 > M3 (0.472 > 0.360):
       Tương tự — context quá phong phú khiến LLM trả lời dài dòng,
       không trực tiếp vào câu hỏi

    Kết luận tổng thể:
    - M2 là lựa chọn tốt nhất trong thực tế (cân bằng Faithfulness + Relevancy)
    - M3 vượt trội về Coverage (Context Recall) nhưng trade-off Faithfulness
    - Cả M2 và M3 đều vượt M1 rõ rệt về mọi mặt

==================================================
TÓM TẮT CÁC KỸ THUẬT MỚI TRONG PHASE 3
==================================================

    BM25 (Sparse Retrieval):
    - Keyword matching, không dùng embedding
    - Công thức BM25Okapi: cải tiến TF-IDF với saturation + length norm
    - Tốt với từ khóa đặc thù, số liệu, tên riêng

    RRF (Reciprocal Rank Fusion):
    - Gộp nhiều ranked list bằng cách dùng rank thay vì score
    - Score = Σ 1/(k+rank+1), k=60 (paper Cormack 2009)
    - Không cần normalize score giữa các retriever khác nhau

    Hybrid Search (M2):
    - Dense (semantic) + BM25 (keyword) → RRF → top-K
    - Bắt được cả paraphrase VÀ keyword exact match

    RAG-Fusion (M3):
    - LLM sinh thêm 3 query variants từ câu hỏi gốc
    - Hybrid Search cho MỖI variant → RRF lần 2
    - Trade-off: Context Recall tăng nhưng Faithfulness giảm

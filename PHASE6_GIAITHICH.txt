--- PHẦN 2: Hàm _clean_text ---

    _VIET_RANGE = "À-ɏḀ-ỿ"
    _CLEAN_RE   = re.compile(r"[^\w\s\.,;:!?()\-–—/\\%" + _VIET_RANGE + "]")
    _SPACE_RE   = re.compile(r"\s+")

    def _clean_text(text: str) -> str:
        text = unicodedata.normalize("NFC", text)
        text = _CLEAN_RE.sub(" ", text)
        text = _SPACE_RE.sub(" ", text)
        return text.strip()

    - Hàm này DUPLICATE với clean_text trong ingest.py vì lý do an toàn:
      Ban đầu backend import clean_text từ ingest.py, nhưng ingest.py
      import langchain.schema → kéo theo langchain_core.memory (deprecated)
      → crash "No module named 'langchain_core.memory'" khi upload PDF.
      Giải pháp: định nghĩa lại inline trong backend, không import từ ingest.

    - _VIET_RANGE = "À-ɏḀ-ỿ": dải ký tự Latin mở rộng và tiếng Việt
        + À-ɏ  (U+00C0 – U+024F): Latin Extended-A/B (dấu phụ châu Âu)
        + Ḁ-ỿ  (U+1E00 – U+1EFF): Latin Extended Additional (tiếng Việt đầy đủ)
      Ghép vào regex character class → bảo vệ ký tự có dấu khỏi bị xóa.

    - Compile regex 1 lần lúc import module (module-level constant),
      không compile mỗi lần gọi hàm → hiệu quả hơn.

--- PHẦN 4: Lazy pipeline loading ---

    _pipelines: dict = {}

    def get_pipeline(method: str):
        if method not in _pipelines:
            if method == "m1":
                from src.methods.m1_naive_rag import NaiveRAG
                _pipelines["m1"] = NaiveRAG()
            elif method == "m2":
                ...
        return _pipelines[method]

    - _pipelines là dict cache global, tồn tại suốt vòng đời server.

    - Tại sao lazy (khởi tạo khi cần, không khởi tạo sẵn)?
      Mỗi pipeline khi khởi tạo:
        + Load HuggingFace embedding model (vài giây, ~120MB RAM)
        + Kết nối ChromaDB
        + Load BM25 index (M2, M3)
        + Load CrossEncoder reranker model (M4, M5)
      Nếu khởi tạo hết 5 cái khi server start → mất 30-60 giây,
      tốn RAM nhiều, trong khi user chỉ dùng 1-2 method.

    - Với lazy loading: server start gần như instant.
      Pipeline chỉ khởi tạo lần đầu khi user chọn dùng.
      Lần thứ 2 trở đi: đã trong cache → trả về ngay.

    - _pipelines.clear() trong /upload: sau khi index PDF mới,
      xóa cache để các pipeline tạo lại với dữ liệu mới.
      (Nếu không clear, pipeline cũ dùng ChromaDB kết nối cũ
      có thể không thấy chunks mới vừa add vào.)


--- PHẦN 5: Endpoint POST /ask ---

    @app.post("/ask")
    def ask(req: AskRequest):
        pipeline = get_pipeline(req.method)
        t0 = time.time()
        result = pipeline.run(req.question)
        latency_ms = round((time.time() - t0) * 1000)
        return {
            "answer": result["answer"],
            "contexts": result.get("contexts", []),
            "latency_ms": latency_ms,
            "method": req.method,
            "method_name": METHOD_NAMES[req.method],
        }

    Request body (JSON):
        { "question": "Tháp Eiffel cao bao nhiêu?", "method": "m3" }

    Response:
        {
          "answer": "Tháp Eiffel cao 324 mét...",
          "contexts": ["Đoạn 1...", "Đoạn 2...", ...],
          "latency_ms": 1843,
          "method": "m3",
          "method_name": "M3 - RAG-Fusion"
        }

    - contexts: danh sách text của các chunk được retrieve (không phải
      toàn bộ ChromaDB, chỉ là top-K chunks pipeline chọn).
      Frontend dùng để hiển thị "Xem nguồn (3 đoạn)" cho user.

    - latency_ms: đo thời gian từ lúc gọi pipeline.run() đến khi có kết quả.
      Không tính thời gian load pipeline lần đầu (lazy init).
      Phản ánh đúng thời gian xử lý thực tế mỗi câu hỏi.


--- PHẦN 6: Endpoint POST /upload ---

    @app.post("/upload")
    async def upload_pdf(file: UploadFile = File(...)):

    Bước 1 — Phát hiện trùng lặp:
        save_path = os.path.join(ABS_PDF_DIR, file.filename)
        is_update = os.path.exists(save_path)

        - Duplicate detection bằng TÊNFILE: nếu đã có file cùng tên

        - Dùng ABS_PDF_DIR (absolute path) để os.path.exists() luôn
          trả về đúng, không phụ thuộc vào working directory của uvicorn.

    Bước 2 — Lưu file:
        content = await file.read()
        with open(save_path, "wb") as f:
            f.write(content)

        - async/await: đọc file upload không blocking, server vẫn
          phục vụ request khác trong lúc chờ đọc xong.
        - Lưu trước, xử lý sau: nếu ingestion fail thì xóa file mới
          (nhưng chỉ xóa nếu is_update=False, tức file chưa tồn tại).

    Bước 3 — Xóa chunks cũ nếu là cập nhật:
        if is_update:
            existing = vs._collection.get(where={"source": file.filename})
            old_ids = existing.get("ids", [])
            if old_ids:
                vs._collection.delete(ids=old_ids)
                chunks_removed = len(old_ids)

        - ChromaDB lưu metadata cho mỗi chunk, trong đó có
          metadata["source"] = tên file PDF gốc.
        - where={"source": file.filename}: query theo metadata để tìm
          tất cả chunks thuộc file này.
        - Xóa hết rồi mới add lại → tránh duplicate chunks.
        - Nếu không xóa: mỗi lần upload lại file là 1 lần nhân đôi số chunks,
          retrieval sẽ trả về nhiều kết quả trùng lặp → giảm chất lượng.

    Bước 4 — Chunk và index:
        loader = PyPDFLoader(save_path)
        pages = loader.load()
        for page in pages:
            page.page_content = _clean_text(page.page_content)
            page.metadata["source"] = file.filename

        splitter = RecursiveCharacterTextSplitter(
            chunk_size=CHUNK_SIZE,
            chunk_overlap=CHUNK_OVERLAP,
            separators=["\n\n", "\n", ".", " "],
        )
        chunks = splitter.split_documents(pages)
        vs.add_documents(chunks)
        _pipelines.clear()

        - Cùng chunk_size và overlap với ingest.py → consistent.
        - Gán metadata["source"] = file.filename: dùng để tìm xóa lần sau.
        - _pipelines.clear(): sau khi index xong xóa pipeline cache.

    Response:
        {
          "status": "updated",   // hoặc "ok" nếu file mới
          "filename": "bao_cao.pdf",
          "chunks_added": 156,
          "chunks_removed": 143  // 0 nếu file mới
        }

--- COMPONENT: QATab ---

    function QATab() {
      const [question, setQuestion] = useState('')
      const [method, setMethod]     = useState('m1')
      const [result, setResult]     = useState(null)
      const [loading, setLoading]   = useState(false)
      const [error, setError]       = useState('')

      const handleAsk = async () => {
        if (!question.trim()) return
        setLoading(true); setError('')
        try {
          const data = await askAPI(question.trim(), method)
          setResult(data)
        } catch (e) {
          setError(e.message)
        } finally {
          setLoading(false)
        }
      }
    }

    - 5 state variables:
        + question : nội dung textarea
        + method   : M1–M5 (default M1)
        + result   : response từ /ask (null khi chưa hỏi)
        + loading  : true khi đang chờ API
        + error    : string lỗi nếu API fail

    - Ctrl+Enter để gửi:
        onKeyDown={e => { if (e.ctrlKey && e.key === 'Enter') handleAsk() }}
      Tiện cho power user không cần dùng chuột.

    - disabled={loading || !question.trim()}: nút "Hỏi" bị disabled khi
      đang loading HOẶC textarea trống (hoặc chỉ khoảng trắng).

    - askAPI() là helper function gọi fetch('/ask', {method:'POST', body:JSON})
      và throw Error nếu res.ok = false.

--- API HELPER FUNCTIONS ---

    async function askAPI(question, method) {
      const res = await fetch('/ask', {
        method: 'POST',
        headers: { 'Content-Type': 'application/json' },
        body: JSON.stringify({ question, method }),
      })
      const data = await res.json()
      if (!res.ok) throw new Error(data.detail || 'Yêu cầu thất bại')
      return data
    }

    - Dùng đường dẫn tương đối '/ask' (không phải 'http://localhost:8000/ask').
      Vite proxy sẽ forward về backend.

    - if (!res.ok) throw new Error(data.detail || ...):
      FastAPI trả lỗi dạng {"detail": "..."} với HTTP 4xx/5xx.
      Nếu res.ok = false (status >= 400) → throw để component catch và
      set error state → hiển thị error message cho user.

    async function uploadAPI(file) {
      const form = new FormData()
      form.append('file', file)
      const res = await fetch('/upload', { method: 'POST', body: form })
      ...
    }

    - FormData: cách chuẩn để upload file qua HTTP.
      Không set Content-Type header thủ công — browser tự set
      "multipart/form-data; boundary=..." với boundary đúng.
      Nếu tự set sẽ miss boundary → FastAPI không parse được file.

--- VẤN ĐỀ 1: No module named 'langchain_core.memory' ---

    Triệu chứng:
        Upload PDF → 500 Internal Server Error
        "No module named 'langchain_core.memory'"

    Nguyên nhân (chuỗi import):
        backend/main.py
          → from src.ingestion.ingest import clean_text
          → ingest.py: from langchain.schema import Document  (deprecated)
          → langchain.schema trigger import langchain_core.memory
          → module này đã bị xóa trong LangChain mới

    Giải pháp:
        1. Fix ingest.py: đổi
               from langchain.schema import Document
           thành
               from langchain_core.documents import Document
        2. Xóa import clean_text từ ingest.py trong backend/main.py,
           định nghĩa lại _clean_text() inline trong backend.
           (Defense-in-depth: backend không phụ thuộc vào import chain của ingest)


--- VẤN ĐỀ 2: Duplicate detection luôn trả về status: "ok" ---

    Triệu chứng:
        Upload cùng 1 file 2 lần → lần 2 vẫn báo "ok" (file mới)
        thay vì "updated".

    Nguyên nhân:
        os.path.exists(save_path) với save_path = "data/pdfs/file.pdf"
        (relative path). uvicorn chạy từ thư mục BTL_RAG/ → resolve đúng.
        Nhưng nếu chạy từ thư mục khác (VD: chạy từ desktop) → path sai
        → file không tìm thấy → is_update = False.

    Giải pháp:
        ABS_PDF_DIR = str(ROOT / PDF_DIR)  (ROOT = absolute path gốc)
        save_path = os.path.join(ABS_PDF_DIR, file.filename)
        is_update = os.path.exists(save_path)


--- VẤN ĐỀ 3: Unicode bị strip trong _clean_text ---

    Triệu chứng:
        Text tiếng Việt sau clean_text bị mất dấu hoặc bị thay bằng khoảng trắng.

    Nguyên nhân:
        Khi Write tool viết string có ký tự Unicode như "À-ÿ" vào file,
        encoding có thể bị sai → các ký tự trong character class bị render
        sai → regex xóa nhầm ký tự tiếng Việt.

    Giải pháp:
        _VIET_RANGE = "À-ɏḀ-ỿ"  (dùng ký tự Unicode thực, không escape)
        _CLEAN_RE = re.compile(r"[^\w\s\.,;:!?()\-–—/\\%" + _VIET_RANGE + "]")
        Compile regex ở module level, verify bằng test Python trực tiếp.


--- VẤN ĐỀ 4: npm không chạy được trong bash terminal ---

    Triệu chứng:
        npm install → "node: command not found"

    Nguyên nhân:
        Node.js được cài trên Windows nhưng PATH của bash (Cygwin/Git Bash)
        không tự động thêm Windows program paths.

    Giải pháp:
        Dùng PowerShell thay vì bash để chạy lệnh npm:
            cd "d:\Ki 8\DL\BTL_RAG\frontend"
            npm install
            npm run dev


--- VẤN ĐỀ 5: Không kill được process uvicorn cũ ---

    Triệu chứng:
        Port 8000 vẫn occupied sau khi tưởng đã tắt.
        Bash: pkill -f uvicorn → không tìm thấy.
        netstat -ano thấy PID 9472 trên port 8000.
        PowerShell: Get-Process -Id 9472 → not found.

    Nguyên nhân:
        Process được spawn từ Cygwin/bash session của Claude.
        PID trong Cygwin khác với PID trong Windows process table.
        PowerShell và netstat thấy Windows PID, không phải Cygwin PID.

    Giải pháp:
        PowerShell: Get-Process python* | Stop-Process -Force
        Kill tất cả Python processes → chắc chắn không còn uvicorn.
        (Chấp nhận risk: kill hết Python process trên máy lúc đó)

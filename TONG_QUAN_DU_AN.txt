PHẦN 1 — ĐỊNH HƯỚNG VÀ MỤC TIÊU BAN ĐẦU
==================================================

MỤC TIÊU TỔNG THỂ:
    Xây dựng hệ thống Hỏi-Đáp (Question Answering) tiếng Việt dựa trên
    tài liệu PDF, sử dụng kỹ thuật RAG (Retrieval-Augmented Generation).
    So sánh và đánh giá định lượng 5 phương pháp RAG từ đơn giản đến
    phức tạp, từ đó rút ra kết luận về kỹ thuật nào hiệu quả nhất.

BÀI TOÁN CỤ THỂ:
    Input : Câu hỏi bằng tiếng Việt + tập tài liệu PDF/văn bản
    Output: Câu trả lời tiếng Việt được hỗ trợ bởi các đoạn văn liên quan
    Đánh giá: So sánh câu trả lời hệ thống với ground truth đã biết trước

5 PHƯƠNG PHÁP RAG (tăng dần độ phức tạp):
    M1 — Naive RAG      : Retrieve top-K bằng vector similarity → LLM sinh đáp án
    M2 — Hybrid Search  : BM25 (keyword) + Dense (semantic) kết hợp qua RRF
    M3 — RAG-Fusion     : LLM sinh thêm 3 câu hỏi biến thể → Hybrid cho từng biến thể → RRF lần 2
    M4 — CRAG           : Sau retrieve, LLM chấm từng chunk CORRECT/AMBIGUOUS/INCORRECT → lọc
    M5 — Self-RAG Lite  : LLM tự quyết định có cần retrieve không → generate → tự kiểm tra → regen nếu cần

ĐỊNH HƯỚNG KỸ THUẬT:
    - Không dùng Streamlit (quá đơn giản), xây React frontend + FastAPI backend riêng
    - Không tự sinh QA benchmark (dễ bị lỗi font PDF tiếng Việt) → dùng ViQuAD 2.0 chuẩn quốc tế
    - Đánh giá 3 lớp: Retrieval (Context Hit Rate), Generation (ROUGE-L, EM), RAG-end-to-end (RAGAS)
    - Ablation study: cô lập từng biến số (chunk size, overlap, top-K, retrieval type, reranking)
    - Toàn bộ embedding chạy local (không tốn API), LLM dùng Groq free tier

CẤU TRÚC DỰ ÁN:
    BTL_RAG/
    ├── data/
    │   ├── pdfs/           ← PDF tiếng Việt (demo/test ingestion)
    │   ├── viquad/         ← 287 context passages từ ViQuAD 2.0
    │   ├── qa_benchmark.json ← 400 cặp QA (dùng để evaluate)
    │   └── chroma_db/      ← Vector store (831 chunks embedded)
    ├── src/

    - Giải pháp: dùng ViQuAD 2.0 — dataset human-verified, text sạch, chuẩn quốc tế

PHẦN 3 — PHASE 2: NAIVE RAG (M1) + EVALUATION FRAMEWORK
==================================================

MỤC ĐÍCH:
    Xây dựng pipeline RAG cơ bản nhất (M1) và toàn bộ framework đánh giá
    để có baseline số liệu tham chiếu cho các method sau.

ĐÃ LÀM GÌ:
    1. DenseRetriever: embed câu hỏi, tìm top-4 chunks gần nhất trong ChromaDB
    2. LLMWrapper: gọi Groq API (llama-3.3-70b-versatile), retry khi rate limit
    3. Prompt template tiếng Việt:
       "Dựa vào ngữ cảnh sau đây, hãy trả lời câu hỏi..."
    4. NaiveRAG pipeline: DenseRetriever → prompt → LLM → answer
    5. Evaluation metrics:
       - Exact Match: so khớp chính xác chuỗi (case-insensitive)
       - ROUGE-L: so trùng khớp n-gram dạng chuỗi dài nhất
       - Context Hit Rate: tỷ lệ câu hỏi có ít nhất 1 chunk chứa đáp án
    6. RAGAS evaluation (50 mẫu): Context Recall, Faithfulness, Answer Relevancy
    7. Script chạy eval với checkpoint mỗi 10 câu (resumable khi bị ngắt)

KẾT QUẢ M1 BASELINE (n=400):
    Exact Match      : 0.1550  (15.5%)
    ROUGE-L          : 0.4527  (45.3%)
    Context Hit Rate : 0.8225  (82.3%)
    Context Recall   : 0.6364  (63.6%, RAGAS n=50)
    Faithfulness     : 0.8667  (86.7%, RAGAS n=50)
    Answer Relevancy : 0.4873  (48.7%, RAGAS n=50)

PHÂN TÍCH M1:
    - CHR 82.5%: retrieval tốt — trong 82.5% câu, ít nhất 1 trong 4 chunks
      chứa đáp án đúng. 17.5% còn lại là retrieval fail hoàn toàn.
    - ROUGE-L 40%: hợp lý cho Naive RAG — LLM hiểu câu hỏi nhưng đôi khi
      diễn đạt khác ground truth (VD: "Tháp Eiffel" vs "tháp Eiffel cao 324m")
    - EM 11%: thấp vì model trả lời thành câu đầy đủ, không extract chính xác
      như extractive QA. Đây là đặc điểm của generative QA, không phải lỗi.
    - Faithfulness 60%: còn 40% câu trả lời có chi tiết không có trong context
      → LLM "bịa" thêm từ kiến thức nội tại (hallucination nhẹ)

VẤN ĐỀ & CÁCH XỬ LÝ:
    - Groq rate limit khi chạy 400 câu liên tục
      → retry backoff: đợi 30s, 60s, 90s... rồi thử lại tự động
    - Checkpoint file không ghi được
      → fix: os.fsync() sau khi json.dump()
    - RAGAS trả về NaN ở run cuối (rate limit khi judge)
      → dùng kết quả RAGAS từ run trước (valid, không bị NaN)

PHẦN 4 — PHASE 3: HYBRID SEARCH & RAG-FUSION (M2, M3)
==================================================

MỤC ĐÍCH:
    Cải thiện retrieval bằng cách kết hợp BM25 (keyword matching) với
    Dense retrieval (semantic matching). Sau đó thử multi-query expansion.

ĐÃ LÀM GÌ:

    M2 — Hybrid Search:
    - BM25Retriever: load toàn bộ 831 chunks từ ChromaDB SQLite,
      build BM25 index trên RAM (rank_bm25 library)
    - HybridRetriever: chạy BM25 và Dense song song, gộp bằng RRF
      Score RRF = 1/(60 + rank_dense) + 1/(60 + rank_bm25)
    - Trả về top-4 chunks có điểm RRF cao nhất

    M3 — RAG-Fusion:
    - LLM sinh 3 câu hỏi biến thể từ câu gốc (khác cách diễn đạt, cùng ý nghĩa)
    - Chạy HybridRetriever cho cả 4 câu (gốc + 3 biến thể)
    - Gộp 4 kết quả retrieval bằng RRF lần 2 → top-4 cuối cùng

KẾT QUẢ SO SÁNH M1 / M2 / M3 (n=400):

    Metric              M1 Naive   M2 Hybrid  M3 Fusion
    ─────────────────────────────────────────────────────
    Exact Match         0.1100     0.1500     0.1575
    ROUGE-L             0.3993     0.4456     0.4462
    Context Hit Rate    0.8250     0.9100     0.9100
    ─────────────────────────────────────────────────────
    Context Recall      0.6667     0.7143     0.8750
    Faithfulness        0.6000     0.7333     0.5294
    Answer Relevancy    0.2473     0.4724     0.3596

PHÂN TÍCH M2:
    - CHR tăng 8.5% (0.825 → 0.910): BM25 bắt được các câu hỏi
      chứa keyword cụ thể mà dense retrieval bỏ sót (VD: tên riêng, số liệu)
    - Faithfulness tăng mạnh nhất: 0.600 → 0.733 (+13.3%)
      Hybrid lấy chunks "đúng" hơn → LLM hallucinate ít hơn
    - Answer Relevancy tăng gần gấp đôi: 0.247 → 0.472
      Context chất lượng cao hơn → câu trả lời tập trung, đúng trọng tâm hơn

PHÂN TÍCH M3:
    - Context Recall cao nhất: 0.875 — multi-query "quét rộng" hơn,
      tìm được nhiều khía cạnh của ground truth hơn
    - Faithfulness thấp hơn M2 (0.529): có nhiều context hơn →
      LLM có xu hướng pha trộn thông tin → tăng hallucination nhẹ
    - Answer Relevancy trung bình (0.360): câu trả lời dài hơn,
      có thể ôm nhiều thông tin hơn mức cần thiết

TỔNG KẾT PHASE 3:
    - M2 là cân bằng tốt nhất (Faithfulness + Relevancy cao)
    - M3 mạnh nhất về Recall nhưng hi sinh Faithfulness
    - Hybrid search là nâng cấp quan trọng nhất cho Retrieval

VẤN ĐỀ & CÁCH XỬ LÝ:
    - ChromaDB SQLite metadata key là 'chroma:document' (không phải 'document')
      → BM25Retriever phải query đúng key khi load chunks
    - Groq daily quota (100K token/ngày) hết khi RAGAS judge M3
      → chạy patch_m3_ragas.py với API key mới

PHẦN 5 — PHASE 4: CRAG & SELF-RAG (M4, M5)
==================================================

MỤC ĐÍCH:
    Thêm lớp "tự kiểm tra" sau retrieval (M4) và trước retrieval (M5),
    xem liệu LLM tự đánh giá có cải thiện chất lượng cuối cùng không.

ĐÃ LÀM GÌ:

    M4 — CRAG (Corrective RAG):
    - Retrieve top-4 chunks bằng Dense
    - LLM chấm từng chunk: CORRECT / AMBIGUOUS / INCORRECT (1 lần gọi)
    - Chunk INCORRECT → bỏ hoàn toàn
    - Chunk AMBIGUOUS → LLM extract câu liên quan (1 lần gọi / chunk)
    - Chunk CORRECT → giữ nguyên
    - Generate câu trả lời từ tập chunks đã lọc
    - Tổng: 2–7 LLM calls / câu hỏi

    M5 — Self-RAG Lite:
    - LLM quyết định: câu hỏi này có cần retrieve không? (YES/NO)
    - Nếu YES → retrieve + generate bình thường
    - LLM tự kiểm tra: câu trả lời có được context hỗ trợ không? (SUPPORTED/NOT_SUPPORTED)
    - Nếu NOT_SUPPORTED → generate lại với prompt chặt hơn
    - Tổng: 3–4 LLM calls / câu hỏi; 47.2% câu bị quyết định KHÔNG retrieve

KẾT QUẢ ĐẦY ĐỦ M1 → M5 (n=400):

    Metric            M1      M2      M3      M4      M5
    ──────────────────────────────────────────────────────
    Exact Match       0.1100  0.1500  0.1575  0.1325  0.1000
    ROUGE-L           0.3993  0.4456  0.4462  0.4030  0.2474
    Context Hit Rate  0.8250  0.9100  0.9100  0.7475  0.4225
    ──────────────────────────────────────────────────────
    Context Recall    0.6667  0.7143  0.8750  0.6250  0.1333
    Faithfulness      0.6000  0.7333  0.5294  0.6825  0.1875
    Answer Relevancy  0.2473  0.4724  0.3596  0.3931  0.2595

PHÂN TÍCH M4:
    - CHR thấp hơn M1 (0.747 vs 0.825): CRAG filter bỏ nhầm một số
      chunk đúng (false negative) — LLM chấm "INCORRECT" dù chunk có đáp án
    - Faithfulness cải thiện (0.683 vs 0.600): chỉ giữ chunk đã được LLM
      xác nhận → câu trả lời bám context tốt hơn
    - ROUGE-L gần bằng M1 (0.403): filter context không giúp cải thiện
      chất lượng câu trả lời nhiều; bottleneck là generation quality
    - Kết luận: CRAG giúp faithfulness nhưng tốn nhiều API calls,
      và có risk loại bỏ nhầm context tốt

PHÂN TÍCH M5 (quan trọng — counterintuitive):
    - Kết quả rất thấp trên benchmark này vì 47.2% câu bị quyết định KHÔNG retrieve
    - Những câu không retrieve → không có context → Context Recall và Faithfulness ≈ 0
    - LLM trả lời từ "kiến thức chung" → không khớp ground truth dựa trên passage cụ thể
    - ROUGE-L 0.247 (thấp nhất): câu trả lời generic không match với text cụ thể trong ViQuAD
    - BÀI HỌC QUAN TRỌNG: Self-RAG thích hợp cho open-domain QA tổng quát
      ("Nhiệt độ sôi của nước là bao nhiêu?" → không cần retrieve)
      Không thích hợp cho passage-based QA ("Theo đoạn văn, X là gì?") vì
      câu trả lời luôn phụ thuộc vào passage cụ thể

BẢO ĐẠT TỐT NHẤT TỪNG METRIC:
    Faithfulness    : M1 Naive RAG (0.867)
    Context Recall  : M3 Fusion (0.875)
    Answer Relevancy: M1 Naive RAG (0.487)
    ROUGE-L         : M1 Naive RAG (0.453)
    Exact Match     : M3 Fusion (0.158)
    → M1 dẫn đầu RAGAS metrics (n=400 đầy đủ); M3 mạnh nhất về Recall và EM
    → M2 vẫn là lựa chọn cân bằng tốt nhất cho production (CHR cao, ổn định)

==================================================
PHẦN 6 — PHASE 5: ABLATION STUDY & ERROR ANALYSIS
==================================================

MỤC ĐÍCH:
    Kiểm tra từng quyết định thiết kế hệ thống: chunk size, overlap, retrieval type,
    top-K, query expansion, reranking. Xác định cái nào thực sự quan trọng.
    Đồng thời phân loại lỗi để hiểu hệ thống fail ở đâu.

ĐÃ LÀM GÌ:
    - 15 experiments (A1–A6), mỗi experiment dùng n=100 câu (seed=42) để tiết kiệm quota
    - A1: Chunking strategy — Fixed vs Recursive, size 200/400/800
    - A2: Chunk overlap — 0% / 10% / 20% (so với baseline 30%)
    - A3: Retrieval type — Dense / BM25 / Hybrid
    - A4: Top-K — 1 / 3 / 4 / 5 / 10
    - A5: Query expansion — Original / Multi-query Dense / RAG-Fusion
    - A6: Reranking — No rerank / Cross-encoder (ms-marco-MiniLM-L-6-v2)
    - Error analysis: phân loại từng câu hỏi vào correct/retrieval_fail/generation_fail/partial/ambiguous

KẾT QUẢ ABLATION:

[A1] Chunking: Recursive+400 tốt nhất
    fixed_200: ROUGE-L=0.314, CHR=0.61  ← chunk nhỏ quá, retrieval kém
    fixed_400: ROUGE-L=0.410, CHR=0.85
    recursive_400: ROUGE-L=0.476, CHR=0.87  ← BEST
    recursive_800: ROUGE-L=0.391, CHR=0.81
    → Recursive splitter tốt hơn Fixed (giữ nguyên câu). Chunk 400 là sweet spot.

[A2] Overlap: càng nhiều càng tốt
    overlap=0  (0%):  ROUGE-L=0.375, CHR=0.81
    overlap=40 (10%): ROUGE-L=0.376, CHR=0.82
    overlap=80 (20%): ROUGE-L=0.404, CHR=0.83
    baseline (30%):   ROUGE-L=0.476, CHR=0.87  ← BEST
    → Overlap 30% (120 chars) đang dùng là optimal. Không overlap → mất câu ở ranh giới chunk.

[A3] Retrieval: Hybrid mạnh về CHR, Dense mạnh về ROUGE-L
    BM25 only: ROUGE-L=0.412, CHR=0.850  ← tệ nhất
    Dense only: ROUGE-L=0.476, CHR=0.870
    Hybrid:     ROUGE-L=0.446, CHR=0.910  ← best CHR
    → BM25 một mình yếu vì tiếng Việt hay có biến thể từ, synonyms.
      Hybrid tăng CHR +4% nhưng ROUGE-L hơi thấp hơn Dense — tradeoff retrieval vs generation.

[A4] Top-K: K=4–5 là sweet spot
    K=1:  ROUGE-L=0.277, CHR=0.61  ← quá ít
    K=3:  ROUGE-L=0.387, CHR=0.79
    K=4:  ROUGE-L=0.476, CHR=0.87  ← BEST ROUGE-L
    K=5:  ROUGE-L=0.469, CHR=0.84
    K=10: ROUGE-L=0.428, CHR=0.91  ← best CHR nhưng ROUGE-L giảm
    → K=10 thêm nhiều chunk nhiễu vào context → LLM "bị phân tâm", ROUGE-L giảm.
      K=4 (đang dùng) là optimal.

[A5] Query expansion: không giúp ích trên ViQuAD
    Original (M1):       ROUGE-L=0.476, CHR=0.870
    Multi-query Dense:   ROUGE-L=0.419, CHR=0.810  ← tệ hơn
    RAG-Fusion (M3):     ROUGE-L=0.446, CHR=0.910
    → Multi-query Dense tệ hơn original — các biến thể câu hỏi lấy thêm noise thay vì signal.
      RAG-Fusion tốt hơn Multi-query nhờ Hybrid search (không phải nhờ query expansion).
      Bài học: trên benchmark passage-based, câu gốc đã đủ tốt để retrieve.

[A6] Reranking:
    No rerank (M2):               ROUGE-L=0.446, CHR=0.910  ← baseline
    Multilingual cross-encoder:   ROUGE-L=0.537, CHR=0.960  ← TỐT NHẤT
    → Model nreimers/mmarco-mMiniLMv2-L12-H384-v1 (train trên 26 ngôn ngữ
      gồm tiếng Việt) cải thiện rõ rệt: +9.1% ROUGE-L, +5% CHR vs không rerank.
    → Đây là kết quả cao nhất trong toàn bộ ablation study.

ERROR ANALYSIS (từ raw results M1–M5):

    Category          M1      M2      M3      M4      M5
    ──────────────────────────────────────────────────────
    correct           58.5%   58.5%   57.2%   52.8%   32.0%
    retrieval_fail    16.8%    7.2%    7.8%   21.0%   56.5%
    generation_fail   11.5%   20.8%   20.5%   16.8%    7.0%
    partial           13.2%   13.5%   14.5%    9.5%    4.5%
    ambiguous_query   25.5%   25.5%   25.5%   25.5%   25.5%

    Ghi chú: tất cả n=400. ambiguous_query tính trên subset câu chứa từ tham chiếu
    (này/đây/đó/trên/dưới...) — câu hỏi mơ hồ không xác định được nguồn.

    Phân tích:
    - M1: main fail = retrieval_fail (16.8%) — dense retrieval còn bỏ sót context
    - M2/M3: generation_fail cao nhất (20.8%, 20.5%) — retrieval tốt hơn M1
      nhưng LLM "bị ngợp" bởi nhiều context chất lượng → khó tổng hợp
    - M4: retrieval_fail tăng vọt (21%) — CRAG filter bỏ nhầm chunk đúng
    - M5: retrieval_fail chiếm 56.5% — chủ yếu do 47.2% câu không retrieve gì
    - 1/4 câu hỏi (25.5%) là ambiguous — hạn chế cố hữu của benchmark,
      không thể cải thiện chỉ bằng kỹ thuật RAG

KẾT LUẬN ABLATION:
    Cấu hình tốt nhất: Recursive, chunk=400, overlap=120, K=4, Hybrid + Multilingual Reranker.
    Reranker multilingual (mmarco-mMiniLMv2) là cải tiến đáng kể nhất (+9.1% ROUGE-L).

PHẦN 8 — TỔNG KẾT & SO SÁNH TOÀN BỘ
==================================================

BẢNG KẾT QUẢ ĐẦY ĐỦ (n=400, UIT-ViQuAD 2.0):

    Metric              M1      M2      M3      M4      M5      Best
    ───────────────────────────────────────────────────────────────────
    Exact Match        0.155   0.150   0.158   0.133   0.100   M3
    ROUGE-L            0.453   0.446   0.446   0.403   0.247   M1
    Context Hit Rate   0.823   0.910   0.910   0.748   0.423   M2/M3
    Context Recall     0.636   0.714   0.875   0.625   0.133   M3
    Faithfulness       0.867   0.733   0.529   0.683   0.188   M1
    Answer Relevancy   0.487   0.472   0.360   0.393   0.260   M1

KẾT LUẬN TỔNG THỂ:

    1. M2 (Hybrid Search) là winner thực tế:
       - Faithfulness và Answer Relevancy cao nhất → câu trả lời đáng tin và đúng trọng tâm
       - CHR cao (91%) → retrieval đáng tin cậy
       - Chi phí hợp lý: 1 LLM call / câu hỏi

    2. M3 (RAG-Fusion) tốt nhất về Recall:
       - Context Recall 0.875 — tìm được context bao phủ ground truth tốt nhất
       - Nhưng Faithfulness thấp (0.529) — nhiều context → LLM dễ hallucinate
       - Chi phí cao hơn: 2 LLM calls / câu hỏi

    3. M4 (CRAG) có ý tưởng hay nhưng hiệu quả thực tế hạn chế:
       - Faithfulness cải thiện so M1 (0.683 vs 0.600)
       - Nhưng CHR giảm vì filter bỏ nhầm (0.748 vs 0.825)
       - Chi phí cao nhất: 2–7 LLM calls / câu hỏi

    4. M5 (Self-RAG) không phù hợp với passage-based benchmark:
       - Điểm thấp nhất ở hầu hết metrics
       - Lý do: benchmark ViQuAD yêu cầu extract từ passage cụ thể
         → tự quyết định không retrieve là sai với loại benchmark này
       - M5 phù hợp hơn cho open-domain QA ("câu hỏi thường thức")

    5. Kỹ thuật nền tảng quan trọng nhất (từ ablation):
       - Chunking: Recursive splitter, size=400, overlap=30% → không thể trade off
       - Top-K: K=4–5 → sweet spot giữa recall và generation quality
       - Hybrid > Dense > BM25 cho tiếng Việt (keyword + semantic)
       - Reranker tiếng Anh không dùng được cho tiếng Việt

GIỚI HẠN CỦA DỰ ÁN:
    - 25.5% câu hỏi trong ViQuAD có từ tham chiếu mơ hồ → không thể giải quyết bằng RAG
    - Cross-encoder reranker chỉ có tiếng Anh → cần tìm model multilingual
    - Groq free tier quota (~100K token/ngày) gây gián đoạn khi evaluate số lượng lớn
    - m1_raw.json chỉ có 161 mục (bị cắt session) → error analysis M1 dùng subset

CẢI TIẾN TIỀM NĂNG:
    - Dùng multilingual cross-encoder (VD: cross-encoder/mmarco-mMiniLMv2-L12-H384)
    - Fine-tune embedding model trên dữ liệu tiếng Việt
    - Cải thiện prompt tiếng Việt với few-shot examples
    - GraphRAG hoặc HyDE cho domain-specific documents

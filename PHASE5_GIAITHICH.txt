MỤC ĐÍCH: Rerank danh sách chunks đã retrieve bằng Cross-Encoder model.
Dùng trong ablation A6 để đánh giá hiệu quả của reranking.

CODE:
    from sentence_transformers import CrossEncoder
    from src.config import TOP_K

    class CrossEncoderReranker:
        def __init__(self):
            self.model = CrossEncoder("cross-encoder/ms-marco-MiniLM-L-6-v2")

        def rerank(self, query: str, docs: list, top_k: int = TOP_K) -> list:
            pairs = [(query, d.page_content) for d in docs]
            scores = self.model.predict(pairs)
            ranked = sorted(zip(scores, docs), key=lambda x: x[0], reverse=True)
            return [d for _, d in ranked[:top_k]]

GIẢI THÍCH:

    Tại sao cần reranker?
    Dense/BM25 retrieval trả về chunks theo embedding similarity hoặc BM25 score.
    Những scores này không phải lúc nào cũng phản ánh đúng "khả năng trả lời câu hỏi".

    Dense: tìm chunk có vector gần với vector câu hỏi
           → tốt với ngữ nghĩa, nhưng không xét mối quan hệ query–document trực tiếp

    Cross-Encoder: nhận vào CẶP (query, document) cùng lúc,
                   xử lý TOÀN BỘ cặp đó trong 1 lần forward pass của BERT
                   → hiểu được mối quan hệ 2 chiều giữa query và document
                   → chính xác hơn Bi-Encoder (Dense) nhưng chậm hơn nhiều

    Tại sao không dùng Cross-Encoder để retrieve trực tiếp?
    Cross-Encoder cần chạy với TỪNG cặp (query, doc) → O(N) forward pass
    Với 831 chunks: 831 inference lần/câu hỏi → quá chậm cho retrieval
    Giải pháp: Retrieve nhanh bằng Dense/Hybrid (top-4K chunks), rồi
    rerank chỉ 4K chunks đó bằng Cross-Encoder → 2-stage pipeline

    CrossEncoder("cross-encoder/ms-marco-MiniLM-L-6-v2"):
    - Model nhỏ (22M params), chạy local, không cần API
    - Trained trên MS MARCO (English Q&A dataset)
    - Giới hạn: trained tiếng Anh → kém hiệu quả với tiếng Việt
      → Đây là lý do A6 cho kết quả kém trong ablation

    rerank(query, docs, top_k):
    - Tạo list các cặp: [(query, chunk1), (query, chunk2), ...]
    - model.predict(pairs): chạy Cross-Encoder trên tất cả cặp,
      trả về score từng cặp (logit/probability)
    - sorted(): sắp xếp theo score giảm dần
    - Trả về top_k docs theo thứ hạng mới

    Trong ablation A6:
    - Retrieve bằng HybridRetriever top-4×3=12 chunks (nhiều hơn usual)
    - Rerank 12 chunks → chọn top-4
    - So sánh với Hybrid không rerank (M2 baseline)

CODE:
    def build_variant_chroma(chroma_dir: str, chunk_size: int,
                             chunk_overlap: int, splitter_type: str = "recursive"):
        db_path = os.path.join(chroma_dir, "chroma.sqlite3")
        if os.path.exists(db_path):
            print(f"  [skip] {chroma_dir} already exists")
            return

        if splitter_type == "fixed":
            splitter = CharacterTextSplitter(
                separator=" ", chunk_size=chunk_size, chunk_overlap=chunk_overlap
            )
        else:  # "recursive"
            splitter = RecursiveCharacterTextSplitter(
                chunk_size=chunk_size, chunk_overlap=chunk_overlap,
                separators=["\n\n", "\n", ".", " "]
            )

        docs = []
        for fname in os.listdir(VIQUAD_DIR):
            loader = TextLoader(os.path.join(VIQUAD_DIR, fname), encoding="utf-8")
            docs.extend(loader.load())

        chunks = splitter.split_documents(docs)
        emb = HuggingFaceEmbeddings(model_name=EMBEDDING_MODEL)
        Chroma.from_documents(chunks, emb, persist_directory=chroma_dir)

    ChromaDB chính (data/chroma_db/) dùng cấu hình cố định: chunk_size=400,
    chunk_overlap=120, splitter=recursive.
    Để so sánh các cấu hình khác, cần tạo ChromaDB mới với chunk khác nhau.

    Mỗi experiment A1/A2 tạo ChromaDB riêng trong data/ablation_chroma/{name}/:

        a1_recursive_200/chroma.sqlite3
        a1_recursive_800/chroma.sqlite3

        a2_overlap_40/chroma.sqlite3

    → Nếu script bị dừng giữa chừng, chạy lại không mất công rebuild

    2 loại splitter:
    "fixed":
        CharacterTextSplitter(separator=" "):
        Cắt tại khoảng trắng, mỗi chunk đúng chunk_size ký tự.
        Không quan tâm cấu trúc văn bản (đoạn, câu).
        → Đơn giản, nhưng có thể cắt giữa câu quan trọng.

        → Thông minh hơn, giữ ngữ nghĩa tốt hơn.

    Ablation chỉ đánh giá trên 100 câu từ qa_benchmark.json (subset).

MỤC ĐÍCH: Phân loại các câu hỏi thất bại trong m1–m5_raw.json vào các
bucket lỗi, không cần gọi LLM (chạy hoàn toàn local).

CODE:

    def classify(item: dict) -> str:
        rl   = rouge_l(item["answer"], item["ground_truth"])
        chr_ = context_hit_rate(item["contexts"], item.get("gold_context", ""))
        if rl >= 0.3:
            return "correct"
        if chr_ == 0.0:
            return "retrieval_fail"
        if chr_ >= 0.3 and rl < 0.1:
            return "generation_fail"
        return "partial"

    def has_ambiguous_ref(question: str) -> bool:
        AMBIGUOUS_WORDS = {"này", "đây", "đó", "kia", "đấy", "ấy",
                           "trên", "sau", "trước", "nó", "họ", "chúng", "nơi", "việc", "điều"}
        tokens = set(question.lower().split())
        return bool(tokens & AMBIGUOUS_WORDS)

GIẢI THÍCH:

    4 BUCKET LỖI:

    "correct" (ROUGE-L >= 0.3):
    Câu trả lời đủ overlap với ground truth → coi là thành công.
    Ngưỡng 0.3 thay vì 1.0 vì model thường trả lời câu đầy đủ
    trong khi ground truth ngắn (VD: "324 mét" vs "Tháp Eiffel cao 324 mét").

    "retrieval_fail" (CHR = 0.0):
    Không có chunk nào trong retrieved contexts chứa gold_context.
    → Retrieval hoàn toàn sai hướng → LLM không có thông tin để trả lời.
    → Giải pháp: cải thiện retrieval (Hybrid, reranker, top-K lớn hơn).

    "generation_fail" (CHR >= 0.3 VÀ ROUGE-L < 0.1):
    Retrieval TÌM ĐƯỢC context đúng (CHR cao) nhưng LLM vẫn trả lời sai (ROUGE-L rất thấp).
    → Retrieval OK, LLM thất bại: hallucinate, không đọc được context, prompt kém.
    → Giải pháp: cải thiện prompt, dùng model lớn hơn.

    "partial" (không khớp cả 3 điều kiện trên):
    ROUGE-L thấp nhưng không hẳn vì retrieval sai hay generation sai.
    Các nguyên nhân: câu trả lời đúng nhưng dùng từ khác (paraphrase),
    ground truth ngắn nhưng answer dài (precision thấp), câu hỏi mơ hồ.

    AMBIGUOUS QUERY — has_ambiguous_ref(question):
    Đếm số câu hỏi có từ tham chiếu như "này", "đó", "kia"...
    Ví dụ: "Công trình này được xây năm nào?" — không có ngữ cảnh trước
    → Model không biết "này" là gì → câu hỏi inherently ambiguous
    Kết quả: ~25.5% câu hỏi trong benchmark có từ tham chiếu.

    analyze_method(method):

    M1: main_fail = partial (13.7%)
    Retrieval tốt (CHR=0.825) nhưng LLM trả lời không khớp hoàn toàn.
    Nguyên nhân: model sinh câu đầy đủ, ground truth ngắn.

    M2/M3: main_fail = generation_fail (~20%)
    Retrieval tốt hơn M1 nhưng generation_fail tăng lên.
    Nghịch lý: context phong phú hơn → LLM có nhiều thông tin → có xu hướng
    sinh câu dài hơn, pha trộn nhiều thông tin → ROUGE-L thấp hơn kỳ vọng.

    M4: main_fail = retrieval_fail (21%)
    CRAG filter bỏ chunks INCORRECT → đôi khi loại nhầm chunk đúng (false negative).
    → CHR giảm từ 0.825 xuống 0.748.

    M5: main_fail = retrieval_fail (56.5%)
    47.2% câu không retrieve gì cả (retrieved=False) → contexts=[]
    → CHR=0 cho những câu đó → bị classify là retrieval_fail.
    Đây không phải lỗi thực sự của M5, chỉ là hiệu ứng của benchmark passage-based.

    ambiguous_query = 25.5% cho tất cả method:
    Cùng 1 câu hỏi → cùng 1 kết quả classify → cùng 25.5%.
    Cho thấy 1/4 câu hỏi trong benchmark có từ tham chiếu không rõ ràng.
